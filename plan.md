# 金融舆情监测方案改进计划

## 1. 当前定位

当前系统已经具备金融舆情监测 POC 的基础能力，包括多数据源爬取、关键词监测方案、字典式情绪分析、舆情检索、预警中心和可视化看板。

但如果目标是通过金融交易社区、行研报告、财经新闻、监管公告等数据，对市场情绪进行较精准判断，并按自定义方案衡量行业或板块的“市场温度”，当前方案还需要从数据、模型、指标、存储和验证闭环几个方面升级。

## 2. 核心问题

### 2.1 情绪判断可信度不足

当前情绪分析主要依赖金融情绪词典，通过正负面词命中数量计算情绪分数。这种方式适合 POC 演示，但在真实金融语境中容易误判。

典型问题：

- 无法准确识别“利空出尽”“短期承压但长期看好”等复杂表达。
- 难以处理标题党、反讽、传闻、复盘、转述和研报委婉措辞。
- 正负面词简单抵消，无法体现语义重点。
- 不区分事实、观点、预测、风险提示和市场传闻。

### 2.2 行业/板块温度尚未成型

目前系统有关键词和实体识别，但还没有完整的行业、板块、概念、股票、指数映射体系，也没有可解释的温度指数公式。

典型缺口：

- 缺少“关键词 -> 股票 -> 行业 -> 概念 -> 指数”的映射链路。
- 缺少行业同义词、上下游产业链、龙头公司和主题概念维护能力。
- 只统计情绪分布和声量，没有综合声量异动、传播热度、来源权重、时间衰减和行情确认。

### 2.3 数据源稳定性和合规风险较高

系统规划的数据源较多，但部分源依赖 Cookie、非公开接口、API Key 或仍是存根实现。

需要重点关注：

- 交易社区和社媒平台反爬策略变化。
- 券商研报、Wind、Bloomberg、Reuters 等数据授权。
- 第三方平台 ToS、robots.txt、版权和个人信息保护要求。
- 非官方接口失效后的降级策略。

### 2.4 数据存储不适合长期分析

当前使用 JSON 文件持久化，适合演示和小规模样本，但不适合持续爬取、历史回测、趋势聚合和高并发检索。

需要支持：

- 内容去重和相似聚类。
- 按时间、来源、行业、实体、方案维度聚合。
- 历史回测和指标追踪。
- 数据质量标记和采集失败追踪。

### 2.5 缺少可量化评估闭环

如果要证明“精准判断市场情绪”，必须有回测和评估机制，而不是只看主观展示效果。

建议评估：

- 情绪指数与行业次日、三日、一周涨跌幅的相关性。
- 温度指数与成交额、换手率、波动率、资金流的相关性。
- 预警命中率、误报率、漏报率。
- 不同数据源、不同模型、不同权重组合的效果对比。

## 3. 目标方案

建议将系统目标拆成三层：

### 3.1 舆情雷达

用于持续发现市场中正在升温或恶化的话题。

能力包括：

- 多源采集。
- 去重聚类。
- 实体识别。
- 风险识别。
- 热点趋势发现。

### 3.2 行业/板块温度计

用于对自定义行业、板块、概念或主题给出 0-100 的温度指数。

建议温度分层：

| 温度区间 | 状态 | 含义 |
| --- | --- | --- |
| 0-30 | 冰点 | 情绪低迷，负面或无人关注 |
| 30-45 | 偏冷 | 关注度和情绪偏弱 |
| 45-60 | 中性 | 情绪和热度处于常态 |
| 60-75 | 偏热 | 正面情绪或讨论热度明显升温 |
| 75-100 | 过热 | 高热度、高波动，需警惕拥挤交易 |

### 3.3 投研辅助信号

用于辅助研究和监控，不直接替代投资决策。

建议输出：

- 温度指数。
- 情绪方向。
- 声量变化。
- 风险等级。
- 置信度。
- 关键驱动新闻或帖子。
- 与行情数据的确认程度。

## 4. 行业/板块温度指数设计

建议采用可解释的综合评分，而不是单一情绪分数。

初版公式：

```text
板块温度 = 情绪得分 * 35%
        + 声量异动 * 20%
        + 传播热度 * 15%
        + 来源可信度 * 15%
        + 行情确认 * 15%
```

### 4.1 情绪得分

来源于文本情绪模型，建议输出 -1 到 1，再归一化到 0-100。

需要区分：

- 强利好。
- 弱利好。
- 中性。
- 弱利空。
- 强利空。
- 风险事件。
- 传闻或未证实信息。

### 4.2 声量异动

衡量当前讨论量相对历史基线的变化。

建议对比：

- 过去 1 小时 vs 过去 7 日同时段均值。
- 过去 6 小时 vs 过去 30 日均值。
- 当日累计声量 vs 近 20 个交易日均值。

### 4.3 传播热度

根据阅读、评论、点赞、转发、收藏等互动指标计算。

不同平台指标需要标准化，避免社媒数据天然压过研报和监管公告。

### 4.4 来源可信度

不同来源应有不同权重。

建议初始权重：

| 来源类型 | 建议权重 |
| --- | --- |
| 监管公告、交易所公告 | 高 |
| 券商研报、机构观点 | 高 |
| 主流财经媒体 | 中高 |
| 交易社区、股吧、雪球 | 中 |
| 短视频、泛社媒 | 中低 |
| 未认证账号、转载内容 | 低 |

### 4.5 行情确认

用市场数据校验舆情温度是否被交易行为确认。

建议指标：

- 板块涨跌幅。
- 成交额变化。
- 换手率。
- 主力资金流。
- 北向资金或 ETF 资金流。
- 波动率。

## 5. 数据能力改进

### 5.1 建立行业映射体系

需要新增或维护以下映射：

- 股票代码、股票名称、简称、别名。
- 股票所属申万/中信/证监会行业。
- 概念板块。
- 指数成分。
- 产业链上下游。
- 主题关键词和同义词。
- 海外映射，例如英伟达对应 AI 算力、半导体、云计算。

### 5.2 建立数据源分级

每个数据源应配置：

- 来源类型。
- 可信度权重。
- 采集频率。
- 授权状态。
- 反爬风险。
- 内容质量评分。
- 是否纳入温度指数。

### 5.3 增强去重和聚类

需要避免同一新闻被多个平台转载后重复放大。

建议：

- URL 去重。
- 标题归一化。
- 文本相似度聚类。
- 同一事件合并为事件簇。
- 事件簇级别计算影响力，而不是单条内容简单累加。

## 6. 模型能力改进

### 6.1 从词典法升级为金融语义模型

建议分阶段升级：

1. 保留词典法作为 baseline。
2. 引入金融文本分类模型，判断利好、利空、中性、风险、传闻。
3. 对高影响内容使用大模型二次分析。
4. 对模型输出增加置信度和原因解释。

### 6.2 增加结构化事件识别

建议识别以下事件类型：

- 政策变化。
- 业绩预告。
- 增减持。
- 并购重组。
- 监管处罚。
- 债务违约。
- 产业景气变化。
- 价格涨跌。
- 供需变化。
- 地缘冲突。
- 研报评级变化。

### 6.3 增加主体相关度评分

一篇内容命中关键词，不代表它真的影响该板块。

需要判断：

- 关键词是否是核心主体。
- 内容是否只是顺带提及。
- 主体是公司、行业、指数还是宏观因素。
- 内容影响方向是否明确。
- 影响是短期交易性还是中长期基本面。

## 7. 产品功能改进

### 7.1 监测方案升级

当前监测方案主要是关键词、来源、阈值。建议扩展为：

- 监测对象：行业、板块、概念、股票池、指数。
- 关键词组：核心词、扩展词、排除词。
- 数据源权重。
- 温度公式配置。
- 预警规则。
- 行情确认规则。
- 输出周期：实时、小时级、日级。

### 7.2 新增行业温度看板

建议看板包含：

- 行业温度排名。
- 温度变化榜。
- 过热行业。
- 冰点行业。
- 情绪趋势。
- 声量趋势。
- 关键驱动事件。
- 代表性内容。
- 行情确认状态。

### 7.3 新增预警类型

建议支持：

- 负面声量突增。
- 风险事件首次出现。
- 温度快速升温。
- 温度过热。
- 情绪与行情背离。
- 研报观点集中转向。
- 社区异常讨论。

## 8. 技术架构改进

### 8.1 存储升级

建议从 JSON 文件升级到数据库。

POC 下一阶段可选：

- SQLite：轻量、迁移成本低，适合本地演示。
- PostgreSQL：适合生产化、多维聚合和长期存储。

推荐表结构方向：

- contents：原始内容。
- content_entities：内容实体。
- content_events：事件识别结果。
- sources：数据源配置。
- monitoring_projects：监测方案。
- industry_mappings：行业映射。
- sentiment_scores：模型评分。
- temperature_snapshots：温度指数快照。
- alerts：预警记录。

### 8.2 任务调度升级

建议支持：

- 分数据源采集频率。
- 失败重试和熔断。
- 采集状态监控。
- 数据源健康度评分。
- 增量采集。
- 采集任务日志。

### 8.3 API 升级

建议新增：

- 行业温度 API。
- 板块温度历史 API。
- 事件簇 API。
- 数据源质量 API。
- 回测评估 API。
- 模型解释 API。

## 9. 回测与评估

### 9.1 建立历史样本集

选择典型板块进行验证：

- AI 科技。
- 新能源。
- 有色金属。
- 半导体。
- 医药。
- 银行。
- 房地产。

### 9.2 评估指标

建议评估：

- 温度指数与未来 1 日、3 日、5 日涨跌幅相关性。
- 温度指数与成交额变化相关性。
- 负面风险预警命中率。
- 重大风险事件提前量。
- 数据源贡献度。
- 模型误判样本分布。

### 9.3 建立人工标注集

建议抽样标注：

- 情绪方向。
- 事件类型。
- 影响主体。
- 影响行业。
- 影响强度。
- 是否为噪声。

人工标注结果用于模型评估和后续训练。

## 10. 实施路线图

### 阶段一：温度指数 MVP

目标：在现有 POC 基础上形成可展示的行业温度能力。

建议工作：

- 新增行业/板块映射配置。
- 为监测方案增加行业或板块维度。
- 实现基础温度公式。
- 新增温度快照数据结构。
- 新增行业温度排名看板。
- 对现有数据做行业聚合。

### 阶段二：数据质量和去重

目标：降低重复转载和低质量内容对温度指数的干扰。

建议工作：

- 实现标题和 URL 去重。
- 增加文本相似聚类。
- 增加数据源可信度配置。
- 增加来源权重。
- 增加时间衰减。
- 增加采集质量监控。

### 阶段三：模型增强

目标：提升情绪和事件判断准确性。

建议工作：

- 引入金融语境分类模型或大模型分析。
- 增加事件类型识别。
- 增加主体相关度评分。
- 增加模型置信度。
- 保留词典命中作为可解释依据。

### 阶段四：行情联动

目标：让温度指数与真实市场行为互相验证。

建议工作：

- 接入行情数据。
- 接入成交额、换手率、资金流等指标。
- 增加情绪与行情背离检测。
- 输出行情确认分。

### 阶段五：回测和生产化

目标：验证指标有效性，并提升系统稳定性。

建议工作：

- 建立历史回测任务。
- 输出行业温度与收益率相关性报告。
- 用数据库替代 JSON 存储。
- 完善权限、日志、任务监控和异常处理。
- 明确数据源授权和合规边界。

## 11. 优先级建议

| 优先级 | 事项 | 原因 |
| --- | --- | --- |
| P0 | 行业/板块映射体系 | 没有映射就无法准确计算行业温度 |
| P0 | 温度指数公式 | 将情绪、声量、来源、行情转成可解释指标 |
| P1 | 数据源权重和去重 | 避免重复转载和低质量来源放大噪声 |
| P1 | 数据库存储 | 支撑历史分析、聚合和回测 |
| P1 | 行业温度看板 | 让产品目标直观呈现 |
| P2 | 金融语义模型 | 提升情绪判断准确度 |
| P2 | 行情联动 | 提升温度指数可信度 |
| P2 | 回测评估 | 证明方案是否真的有效 |

## 12. 总体判断

当前方案方向正确，适合作为金融舆情雷达和市场情绪看板的原型。但如果要达到“精准判断市场情绪”和“按行业/板块衡量市场温度”的目标，下一阶段重点不应继续单纯增加爬虫数量，而应优先建设：

1. 行业/板块映射体系。
2. 可解释的温度指数。
3. 数据源权重和去重。
4. 金融语义情绪模型。
5. 行情联动验证。
6. 历史回测评估。

只有形成“采集 -> 识别 -> 聚合 -> 指数 -> 预警 -> 回测”的闭环后，这套系统才具备从 POC 走向可用投研辅助工具的基础。
